# ThreadNotes — Deployment Guide

A **SaaS split**: the FastAPI backend (this repo's `backend/`) is deployed once to
Render as a Docker service; the Electron desktop client (`frontend/`) is built into
an installer that ships to users and talks to that cloud backend. **No API keys ever
ship in the client.**

---

## 1. Backend → Render (Docker)

The backend deploys from [`render.yaml`](render.yaml) using
[`backend/Dockerfile`](backend/Dockerfile) (Python 3.11-slim + `ffmpeg`).

### 1a. First deploy
1. Push this repo to GitHub/GitLab.
2. In Render: **New → Blueprint**, select the repo. Render reads `render.yaml` and
   provisions the `threadnotes-backend` web service automatically.
3. Render builds the Docker image (installs `ffmpeg`, then `requirements.txt`).
4. You'll get a public URL like `https://threadnotes-backend.onrender.com`.

### 1b. Environment variables (set in the Render dashboard)
Every secret in `render.yaml` is marked `sync: false`, so Render **prompts for it on
first deploy** (or set later under **Service → Environment**). Nothing secret is
committed to the repo. Set these values:

| Variable | Notes |
|---|---|
| `COSMOS_ENDPOINT` | `https://<account>.documents.azure.com:443/` |
| `COSMOS_KEY` | Cosmos primary key |
| `COSMOS_DATABASE` | e.g. `threadnotes-db` |
| `AZURE_SPEECH_KEY` | Azure Speech resource key |
| `AZURE_SPEECH_REGION` | e.g. `swedencentral` |
| `AZURE_OPENAI_KEY` | Azure OpenAI key |
| `AZURE_OPENAI_ENDPOINT` | `https://<resource>.openai.azure.com/` |
| `RESEND_API_KEY` | Resend API key (`re_...`) — email delivery for OTP/reset |
| `EMAIL_SENDER` | Resend-verified **from** address (see §3a) |

Already defaulted in `render.yaml` (override only if needed): `JWT_SECRET`
(auto-generated), `COSMOS_USERS_CONTAINER=users`, `COSMOS_TRANSCRIPTS_CONTAINER=transcripts`,
`AZURE_OPENAI_API_VERSION`, `AZURE_DIARIZE_DEPLOYMENT`.

> ⚠️ `JWT_SECRET` is auto-generated by Render. On first deploy this invalidates any
> previously issued login tokens — expected. Don't change it later or you'll log
> everyone out.

### 3a. Email delivery — Resend (HTTP API, not SMTP)
Render's free tier **blocks outbound SMTP (port 587)** — `smtplib` fails with
`[Errno 101] Network is unreachable`. So OTP / password-reset emails are sent via
the **Resend HTTP API** over HTTPS (`send_otp_email` in `backend/main.py`, using the
already-installed `httpx` — no extra dependency).

Two env vars (both `sync: false` in `render.yaml`):

| Variable | Notes |
|---|---|
| `RESEND_API_KEY` | From the Resend dashboard (`re_...`). |
| `EMAIL_SENDER` | The `from` address. **Must be a Resend-verified sender.** Falls back to `onboarding@resend.dev` if unset. |

> ⚠️ **Domain verification is the gotcha.** Resend only delivers from a verified sender:
> - **Testing:** leave `EMAIL_SENDER` unset (uses `onboarding@resend.dev`) — but Resend
>   will only deliver to **the email you signed up to Resend with**. OTPs to any other
>   address are silently dropped.
> - **Production:** verify your domain in Resend (add their DNS records), then set
>   `EMAIL_SENDER=noreply@yourdomain.com`. Only then can you email arbitrary users.

The old Gmail SMTP vars (`EMAIL_PASSWORD`, `SMTP_HOST`, `SMTP_PORT`) are **no longer
used** — remove them from Render if present. Watch the logs for
`[email] sent OK -> … (id=…)`; a `Resend error 4xx` line quotes the exact reason
(unverified domain, bad key, etc.).

### 1c. Local image test (optional, needs Docker)
```bash
docker build -t threadnotes-backend ./backend
docker run -p 8000:8000 --env-file backend/.env threadnotes-backend
# open http://localhost:8000/docs
```

## 2. Cosmos DB — network / firewall rules

Render's containers reach Cosmos over the public internet, so Cosmos must allow it:

- **Azure Portal → your Cosmos account → Networking.**
- Simplest (testing): **Allow access from All networks**.
- Tighter (production): keep **Selected networks** and add Render's **outbound IP
  addresses** (Render dashboard → service → **Connect / Outbound IPs**) to the
  Cosmos firewall allow-list. Re-check these if you change Render region/plan.
- Ensure the **database and containers exist** (`users`; `transcripts` is created on
  first diarization save by the Cloud Vault's `/diarize/stream`).

If the backend logs Cosmos timeouts/403s on startup or login, this firewall is the
usual cause.

## 3. CORS — the `app://local` requirement

The **packaged Electron renderer's `Origin` is `app://local`** (not `localhost`), and
the dev renderer is `http://localhost:3000`. The backend must allow both.

Current `backend/main.py` uses `allow_origins=["*"]`, which works for the
**non-credentialed** requests the app makes today (Bearer header in JS, no cookies).
When you're ready to tighten it, pin explicitly:

```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=["app://local", "http://localhost:3000"],
    allow_credentials=True,            # only meaningful if you switch to cookies
    allow_methods=["GET", "POST", "OPTIONS"],
    allow_headers=["Authorization", "Content-Type"],
)
```

> Note: `allow_origins=["*"]` together with `allow_credentials=True` is rejected by
> browsers — if you ever move to cookie auth, you **must** list explicit origins.

## 4. Frontend → Electron installer (build-time URL baking)

**Critical:** `NEXT_PUBLIC_API_URL` is a Next.js `NEXT_PUBLIC_*` variable. It is
**inlined into the static bundle at `next build` time — NOT read at runtime.** So the
backend URL is frozen into the `.exe` when you build it. Changing it later means
**rebuilding and redistributing** the installer.

### Steps to ship a client pointed at production
1. Edit [`frontend/.env.production`](frontend/.env.production):
   ```
   NEXT_PUBLIC_API_URL=https://threadnotes-backend.onrender.com
   ```
   (your real Render URL, HTTPS, no trailing slash).
2. Build the installer:
   ```bash
   cd frontend
   npm run dist:win
   ```
   `dist:win` runs `next build` (which applies `.env.production`, overriding `.env`)
   then `electron-builder`. Output: `frontend/dist-electron/ThreadNotes Setup x.x.x.exe`.
3. Distribute that `.exe`.

| Build | Env file used | API target |
|---|---|---|
| `npm run electron:dev` | `.env` | `http://localhost:8000` |
| `npm run dist:win` | `.env.production` | your Render URL |

> If a shipped client is hitting the wrong backend, it was built with the wrong
> `.env.production` — rebuild; there is no runtime override.

## 5. Post-deploy smoke test
1. `https://<render-url>/docs` loads (FastAPI Swagger) → service is up.
2. From the desktop app: sign up / log in (exercises Cosmos + JWT + email OTP).
3. Start a recording → confirm `GET /azure/token` returns 200 (Azure Speech token).
4. Finish a recording → confirm `POST /diarize/stream` returns segments.

---

## 6. Operational notes
- **Free plan sleeps** after ~15 min idle; first request cold-starts in ~30–60s,
  which can stall the first `/azure/token` or recording start. Use the **Starter**
  plan for real users.
- **Token endpoint is auth-protected** — clients need a valid app JWT; the
  `AuthInterceptor` auto-redirects to `/auth` on any backend `401`.
- **ffmpeg** is provided by the Docker image (the `/transcribe` upload path shells
  out to it) — no extra Render config needed.
